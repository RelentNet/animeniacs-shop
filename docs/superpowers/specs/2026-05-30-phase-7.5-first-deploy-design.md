## 0. Goals

Take the code that already works on localhost and put it on the internet
at a staging URL. End state: `https://dev.animeniacs.shop` serves the
Phase 7 code path against the Square sandbox; the deferred 12-step
Phase 7 smoke checklist runs green; you can share the URL with people
and they can navigate, add to cart, and complete a sandbox purchase.

This phase ships zero new product features. It is **pure deployment
infrastructure work**, justified by:
- Every phase from here forward builds on undeployed code; the
  first-deploy blast radius grows with each phase.
- The Phase 6 "white text on white background" pain and the Phase 7
  sandbox-smoke deferral are both symptoms of the same gap: no staging
  environment. Closing the gap unblocks every subsequent smoke story.
- The Phase 7 handoff already contains a 9-step deploy task list. This
  phase implements it with the small additions needed (migration step
  + compose profile shape).

## 1. Non-goals (Phase 7.5 does NOT ship)

- **No production deployment.** Sandbox-only. Production cutover is a
  future phase (probably its own Phase 7.6 or rolled into a launch
  phase) after operator confirms sandbox staging smoke is green for
  some period of time.
- **No new product features.** No promo bar, no abandoned-cart emails,
  no refund notifications. Those are Phase 8+ candidates.
- **No CI/CD pipeline beyond Coolify's built-in auto-deploy.** Push to
  `main` → Coolify rebuilds. No GitHub Actions, no automated lint/test
  on push (operator runs those locally before pushing).
- **No custom monitoring / observability stack.** Coolify dashboard +
  Plausible (already running) + Discord webhook (Phase 7 already wired)
  are the v1 observability layer.
- **No multi-environment setup.** Sandbox-only deploy. No "staging vs
  prod" infrastructure split yet.
- **No backup / disaster-recovery procedure.** Coolify Postgres has a
  daily-backup option in its dashboard; turning it on is operator-side.
  Documented but not gated.
- **No load testing.** The site is staging-only with one user (you).
- **No SSL certificate management.** Coolify + Let's Encrypt handles
  this automatically via Traefik.

## 3. Locked decisions

### Operator-approved during brainstorm

1. **Phase 8 scope = Phase 7.5 first-deploy (Path A).** Not a feature phase. Operator picked this over feature work specifically to unblock future-phase smoke stories.

### Self-locked by master terminal

2. **Migration strategy: init container in compose.** Add a `migrate` service in `compose.yml` that runs `pnpm db:migrate` once before `app` starts. Uses the deps stage's image (which has pnpm + drizzle-kit). Idempotent: re-running on an up-to-date database is a no-op. This survives every future migration without compose edits.

3. **Git source binding: public repository.** Make `itkujo/animeniacs-shop` public. The repo contains zero secrets (all in `.env.local` / Coolify env, gitignored). Lowest friction. If operator strongly prefers private, the fallback is the existing Coolify GH App `helpless-hippopotamus-ogo0o4g0` — but that requires Coolify to be installed on the repo at the GitHub side, which is an extra step.

4. **Postgres: separate Coolify Postgres resource (NOT embedded in compose for deploy).** Coolify dashboard creates a managed Postgres resource. Its connection string is set as `DATABASE_URL` in the app's Coolify env. The compose `postgres` service uses the `local` profile so it runs for local dev but NOT in Coolify deploys. Documented in this spec and the migration command in the `migrate` service.

5. **Compose profiles for environment-specific services.** `compose.yml` keeps the embedded `postgres` under `profiles: [local]`. Coolify (which doesn't pass `--profile local`) skips it. The `migrate` service runs in both contexts but is idempotent. Single source of truth; no `compose.coolify.yml` drift risk.

6. **Domain: `dev.animeniacs.shop`.** DNS is already configured per Phase 4 handoff (`5.161.88.222` points at the Coolify host). Coolify Traefik handles routing + SSL.

7. **Phase tag: `phase-7.5-first-deploy`.** Decimal version makes the chain explicit. The next Phase 8 brainstorm continues from this tag.

8. **No new unit or integration tests in Phase 7.5.** The phase is infrastructure. The "acceptance" is the deferred 12-step Phase 7 smoke running green. Existing 253/253 unit + 75/75 integration must still pass after any compose / Dockerfile changes.

9. **Healthcheck route already exists** at `src/app/api/health/route.ts`. Verify it returns 200 on the deployed URL. No new route needed.

10. **First-deploy ordering: deploy-then-webhook-subscription.** The Square sandbox webhook subscription URL `https://dev.animeniacs.shop/api/webhooks/square` must respond before the subscription can be created (Square does a verification ping). So: deploy first (with `SQUARE_WEBHOOK_SIGNATURE_KEY` blank, webhook handler will 500 on incoming traffic — fine), then create subscription, copy key to Coolify env, redeploy. Documented in the plan as a two-redeploy flow.

11. **New Logto application for `dev.animeniacs.shop`.** A separate registration in the Logto admin console at `auth-admin.animeniacs.shop`, redirect URL `https://dev.animeniacs.shop/callback`. New `LOGTO_APP_ID` + `LOGTO_APP_SECRET` for that app. Localhost `.env.local` keeps its own values; Coolify env gets the new ones. Both apps live in the same Logto instance — they're sibling applications.

12. **`content:build` step in Dockerfile remains as-is.** Phase 1's content manifest build runs during the `builder` stage. No change needed.

1. Public GitHub repository binding works (or operator-chosen alternative).
2. Coolify application `animeniacs-shop-dev` exists in the `website` project, sources from `https://github.com/itkujo/animeniacs-shop` `main`, builds via dockercompose.

   - Site: `NEXT_PUBLIC_SITE_URL=https://dev.animeniacs.shop`, `NODE_ENV=production`
   - Logto: new app's `LOGTO_APP_ID`, `LOGTO_APP_SECRET`, `LOGTO_COOKIE_SECRET`, `LOGTO_ENDPOINT`

6. Smoke 1 — healthcheck: `curl -s https://dev.animeniacs.shop/api/health` returns 200.
7. Smoke 2 — API contract: `curl -s -X POST https://dev.animeniacs.shop/api/checkout -d '{}'` returns 400. `curl -s -X POST https://dev.animeniacs.shop/api/webhooks/square -d '{}'` returns 401.
8. Square sandbox webhook subscription created pointing at the dev URL, key set in Coolify, redeployed.
9. Phase 7 12-step manual smoke checklist (from plan §G.3) runs green end-to-end on `dev.animeniacs.shop`.
10. Operator updates `phase-07-handoff.md` "Verification state at handoff" section with a one-line confirmation that smoke passed (date + result).
11. `pnpm lint`, `pnpm typecheck`, `pnpm test`, `pnpm test:integration`, `pnpm build` still pass locally (no regression from compose / Dockerfile edits).
12. Tag `phase-7.5-first-deploy` applied.

## 6. compose.yml edits

The current `compose.yml`:
- Has `postgres` and `app` services
- `app` healthcheck wired
- App env vars: `NODE_ENV`, `DATABASE_URL`, `NEXT_PUBLIC_SITE_URL`, `LOGTO_*`

What Phase 7.5 adds:

```yaml
services:
  postgres:
    # ...existing config unchanged...
    profiles: [local]    # NEW: only runs with `docker compose --profile local up`
                          # Coolify omits the flag and skips this service.

  migrate:
    build:
      context: .
      dockerfile: Dockerfile
      target: migrate-runtime    # new stage; see §7 below
    container_name: animeniacs-migrate
    environment:
      DATABASE_URL: ${DATABASE_URL}
    command: ["pnpm", "db:migrate"]
    restart: "no"          # one-shot
    # In local profile, depend on the embedded postgres.
    # In Coolify, the operator-side Coolify Postgres is reachable via
    # DATABASE_URL; no depends_on needed (Coolify ensures the resource
    # is provisioned before the app stack starts).
    depends_on:
      postgres:
        condition: service_healthy
        required: false    # makes Coolify (with no embedded postgres) happy

  app:
    # ...existing config...
    environment:
      NODE_ENV: production
      DATABASE_URL: ${DATABASE_URL}    # MODIFIED: now reads from env, not hardcoded
      NEXT_PUBLIC_SITE_URL: ${NEXT_PUBLIC_SITE_URL:-http://localhost:3000}
      LOGTO_ENDPOINT: ${LOGTO_ENDPOINT:-https://auth.animeniacs.shop}
      LOGTO_COOKIE_SECRET: ${LOGTO_COOKIE_SECRET}
      LOGTO_APP_ID: ${LOGTO_APP_ID:-}
      LOGTO_APP_SECRET: ${LOGTO_APP_SECRET:-}
      # NEW Phase 7 / Phase 7.5 env vars:
      SQUARE_ENV: ${SQUARE_ENV:-sandbox}
      SQUARE_ACCESS_TOKEN: ${SQUARE_ACCESS_TOKEN}
      SQUARE_LOCATION_ID: ${SQUARE_LOCATION_ID}
      SQUARE_WEBHOOK_SIGNATURE_KEY: ${SQUARE_WEBHOOK_SIGNATURE_KEY:-}
      DISCORD_ORDER_WEBHOOK_URL: ${DISCORD_ORDER_WEBHOOK_URL:-}
      SMSGATE_USER: ${SMSGATE_USER:-}
      SMSGATE_PASS: ${SMSGATE_PASS:-}
      SMSGATE_BASE_URL: ${SMSGATE_BASE_URL:-https://sms.relentnet.dev}
      NEXT_PUBLIC_PLAUSIBLE_SCRIPT_URL: ${NEXT_PUBLIC_PLAUSIBLE_SCRIPT_URL:-}
      NEXT_PUBLIC_PLAUSIBLE_DOMAIN: ${NEXT_PUBLIC_PLAUSIBLE_DOMAIN:-animeniacs.shop}
    depends_on:
      migrate:
        condition: service_completed_successfully
      postgres:
        condition: service_healthy
        required: false
```

Compatibility notes:
- `required: false` on `depends_on` is Docker Compose v2.20+ syntax. Coolify's compose runner supports it (verified via the AGENTS.md global "Coolify deploys" note context).
- Locally, `docker compose up --profile local` runs Postgres + migrate + app. Without `--profile local`, only migrate + app run, which would fail locally (no Postgres). To avoid breaking local dev workflow, the plan adds a small note to README / operator runbook: **for local dev, use `--profile local`** OR keep using `pnpm dev` against the standalone Postgres container started separately (which is what operator already does).

## 7. Dockerfile edits

The migration sidecar uses the `deps` stage. That stage doesn't have
the source code, only `node_modules` + `package.json`. To run
`drizzle-kit migrate`, we need:
- `drizzle.config.ts`
- `drizzle/migrations/*.sql`
- `src/lib/db/schema.ts` (drizzle-kit reads this)

These exist in the build context. Two options:

**Option (a):** Have the `migrate` service mount the build context at runtime (volume mount). Simplest, but Coolify's compose pipeline doesn't easily volume-mount the build context.

**Option (b):** Add a new Dockerfile stage `migrate-runtime` that COPYs the necessary files into a deps-based image. Then the compose `migrate` service uses `target: migrate-runtime`.

Going with **(b)**. New Dockerfile addition:

```dockerfile
# --- Stage 4: migration runner ---
# A one-shot image used by the `migrate` compose service.
# Has pnpm + node_modules + drizzle-kit + the source files drizzle-kit
# needs to read.
FROM deps AS migrate-runtime
WORKDIR /app
# Copy only what drizzle-kit needs to run migrations:
COPY drizzle.config.ts ./
COPY drizzle ./drizzle
COPY src/lib/db ./src/lib/db
COPY tsconfig.json ./
# pnpm + drizzle-kit are in the inherited node_modules.
CMD ["pnpm", "db:migrate"]
```

Compose then uses `target: migrate-runtime` instead of `target: deps`.
